## 1. 项目概述

### 1.1 项目背景

心智障碍群体（包括自闭症谱系障碍、智力障碍、认知障碍、情绪障碍等）在日常沟通、情绪表达与需求传递方面面临显著困难。现有辅助工具往往缺乏个性化评估与实时反馈能力，无法满足该群体的真实需求。

### 1.2 项目目标

构建一套**在线心理评估与行为能力分析辅助支持系统**，实现以下核心目标：

| 目标编号 | 目标描述 | 衡量指标 |
|---------|---------|---------|
| G-01 | 提供标准化心理评估量表在线化 | 覆盖≥15种评估量表 |
| G-02 | 基于AI的行为模式识别与分析 | 识别准确率≥85% |
| G-03 | 个性化沟通辅助与情绪表达支持 | 用户满意度≥80% |
| G-04 | 多维度能力成长追踪 | 数据可视化覆盖率100% |
| G-05 | 家属/照护者协同管理 | 支持≥3种角色权限 |
| G-06 | 无障碍访问体验 | 通过WCAG 2.1 AA级认证 |

### 1.3 项目范围

**包含范围：**
- Web端管理后台（评估管理、用户管理、数据分析）
- 移动端应用（iOS/Android，用户侧使用）
- AI评估引擎（行为分析、情绪识别、自然语言处理）
- 评估量表引擎（可配置量表、自动评分、报告生成）
- 数据可视化与报表系统
- 家属/照护者协作门户

**不包含范围：**
- 医疗诊断功能（系统定位为辅助支持，非诊断工具）
- 硬件设备研发
- 线下服务运营

### 1.4 关键约束

- **合规性：** 需符合《个人信息保护法》《数据安全法》《精神卫生法》及HIPAA相关条款
- **特殊人群：** 所有交互设计必须满足无障碍标准（WCAG 2.1 AA）
- **数据安全：** 心理评估数据属敏感个人信息，需端到端加密存储与传输
- **交付周期：** 自项目启动之日起 10 个月内完成V1.0上线

## 3. 开发方法论

### 3.1 敏捷 + DevSecOps 融合模式

采用 **Scrum + DevSecOps** 融合开发模式，兼顾迭代灵活性与安全合规要求。

```
┌─────────────────────────────────────────────────────────┐
│                    敏捷迭代 (2周/Sprint)                   │
│                                                         │
│  Sprint规划 → 每日站会 → 开发 → 代码审查 → 自动化测试      │
│       ↓                                              ↓  │
│  持续集成 ──→ 安全扫描 ──→ 自动化部署 ──→ 监控反馈         │
│                                                         │
│                    ←──── 持续改进 ────→                    │
└─────────────────────────────────────────────────────────┘
```

### 3.2 核心实践

| 实践 | 描述 | 工具 |
|-----|------|-----|
| **迭代周期** | 2周Sprint，每月一次发布窗口 | Jira |
| **需求管理** | 用户故事映射 + MoSCoW优先级 | Jira + Confluence |
| **代码审查** | 强制PR审查，至少2人approve | GitHub/GitLab |
| **CI/CD** | 提交即构建、合并即部署（staging） | GitHub Actions / ArgoCD |
| **安全左移** | 代码提交即触发SAST/SCA扫描 | SonarQube + Snyk |
| **自动化测试** | 单元测试覆盖率≥80%，E2E核心流程100% | Jest + Playwright |
| **基础设施即代码** | 所有环境通过IaC管理 | Terraform + Helm |
| **可观测性** | 全链路追踪、结构化日志、业务指标 | Prometheus + Grafana + ELK |

### 3.3 安全合规集成 (DevSecOps)

```
Developer → Commit → SAST/SCA Scan → Merge → Build → DAST → Deploy → Runtime Security
              │                          │         │         │
          SonarQube                 Container     OWASP    Falco
          Snyk                       Scan         ZAP      WAF
```

- **SAST（静态应用安全测试）：** 每次PR触发
- **SCA（软件组成分析）：** 每次依赖变更触发
- **DAST（动态应用安全测试）：** Staging环境每次部署触发
- **容器镜像扫描：** 每次构建触发，高危漏洞阻断发布
- **运行时安全：** Falco实时监控容器异常行为

### 5.7 双MQ架构说明

系统采用 **Kafka + RabbitMQ** 双消息中间件架构，各司其职、互补协作：

| 维度 | Kafka | RabbitMQ |
|------|-------|----------|
| **定位** | 事件流平台 | 任务队列中间件 |
| **核心用途** | 行为事件流、评估事件溯源、审计日志持久化 | 异步任务分发、评估调度、通知推送 |
| **数据特性** | 高吞吐、持久化、可回溯 | 低延迟、精准路由、优先级队列 |
| **消费模式** | 发布/订阅（多消费者组） | 点对点 / 发布订阅 |
| **典型场景** | 行为数据采集→实时分析管道；评估完成→多下游消费 | 评估报告异步生成；邮件/短信/推送通知分发 |
| **顺序保证** | 分区内严格有序 | 队列内FIFO |
| **消息保留** | 基于时间/容量策略保留 | 消费确认后删除 |

```
┌──────────────┐     行为事件/评估事件      ┌──────────────┐
│  数据采集层   │ ──────────────────────→  │    Kafka      │
│ (移动端/Web)  │                           │  (事件流)     │
└──────────────┘                           └──────┬───────┘
                                                  │
                                    ┌─────────────┼─────────────┐
                                    ↓             ↓             ↓
                            实时分析服务    审计日志服务    事件溯源存储
                                                  
┌──────────────┐     异步任务指令           ┌──────────────┐
│  业务服务层   │ ──────────────────────→  │   RabbitMQ    │
│ (评估/通知)   │                           │  (任务队列)   │
└──────────────┘                           └──────┬───────┘
                                                  │
                                    ┌─────────────┼─────────────┐
                                    ↓             ↓             ↓
                            报告生成Worker  通知推送Worker  评估调度Worker
```

### 6.3 Code Review 规范

- **审查清单：**
  - [ ] 功能正确性
  - [ ] 代码可读性
  - [ ] 安全漏洞检查
  - [ ] 单元测试覆盖率
  - [ ] 无障碍合规性
  - [ ] 性能影响评估
  - [ ] 敏感信息泄露检查
- **SLA：** PR应在 **24小时内** 完成审查
- **自动化门槛：** 所有CI检查必须通过（测试、Lint、安全扫描）

Release:

#### 10.2.2 文档分类与更新机制

| 文档类型 | 更新频率 | 责任人 | 审核人 | 时效要求 |
|---------|---------|-------|-------|---------|
| 架构设计文档 | 重大变更时 | Tech Lead | 后端/前端架构师 | 变更后3个工作日内更新 |
| API接口规范 | 每次接口变更时 | 后端开发 | 后端架构师 | PR合并前同步更新 |
| 运维手册 | 每次部署变更时 | DevOps | Tech Lead | 部署前更新完毕 |
| 项目进度报告 | 每Sprint末 | PM | Steering Committee | Sprint Review后1个工作日 |
| 技术调研报告 | 按需 | 调研人 | Tech Lead | 调研完成后2个工作日 |
| 故障处理手册 | 每次P0/P1故障后 | 值班工程师 | DevOps | 故障复盘后1个工作日 |

**文档质量标准：**
- 所有文档必须包含：创建日期、最后更新日期、Owner、版本号
- 代码相关文档必须与代码仓库双向关联（Confluence链接到Git文件，Git README链接到Confluence）
- 过期文档（超过3个月未更新）自动标记为"待审核"状态

### 10.3 技术分享制度

| 制度 | 频率 | 时长 | 参与人 | 内容范围 | 产出物 |
|-----|------|------|-------|---------|-------|
| **Tech Talk** | 双周一次（周五下午） | 45分钟（30分钟分享+15分钟Q&A） | 全体技术团队 | 新技术调研、项目技术实践、踩坑经验、开源项目解读 | 录屏+PPT存入知识库 |
| **Architecture Review** | 月度一次 | 90分钟 | 架构师+Tech Lead+模块Owner | 系统架构演进讨论、技术债务梳理、性能优化方案、安全架构评审 | 评审纪要+行动项跟踪 |

**Tech Talk排班规则：**
- 按团队轮值：前端→后端→AI/ML→DevOps→QA，循环排期
- 每人每季度至少分享1次
- 新入职成员在第3-4周进行"模块介绍"分享
- 分享主题提前1周在Confluence公示，参会者可提前准备问题

**Architecture Review议程：**
1. 上月行动项回顾（15分钟）
2. 当前架构痛点与瓶颈分析（30分钟）
3. 架构演进方案讨论（30分钟）
4. 下月行动项确认（15分钟）

### 10.4 关键岗位AB角制度

为确保项目关键岗位的连续性，降低人员变动风险，实行AB角制度：

| 岗位 | A角（主责） | B角（备岗） | B角履职要求 |
|------|-----------|-----------|-----------|
| 后端架构师 | 后端架构师 | 后端开发高级工程师×1 | 每月参与架构评审；熟悉核心架构决策与设计文档 |
| 前端架构师 | 前端架构师 | 前端开发高级工程师×1 | 每月参与架构评审；掌握前端技术栈与构建配置 |
| AI/ML Lead | AI/ML Lead | AI/ML高级工程师×1 | 参与模型评审；掌握模型训练流程与推理服务架构 |
| DevOps | DevOps工程师 | 后端开发×1 | 熟悉CI/CD流水线配置；掌握K8s集群运维基础操作 |
| 安全工程师 | 安全工程师 | DevOps工程师 | 熟悉安全扫描工具配置；掌握安全事件响应SOP |
| PM | 项目经理 | 产品负责人 | 参与每周项目状态会；掌握里程碑与风险清单 |

**AB角履职机制：**
- B角每月至少参与1次A角的核心工作场景（如架构评审、安全扫描、部署操作）
- A角休假或离岗时，B角自动接替职责，无需额外交接
- 每季度进行1次AB角切换演练，验证B角可独立履职
- 关键文档（架构决策、运维SOP、安全方案）必须由A角和B角共同Review

| 移动端 | iOS 15+ / Android 12+兼容 | 兼容性测试报告 |
| 通知系统 | Push/短信/邮件三通道 | 功能演示 |

| 验收项 | 标准 | 验证方式 |
|-------|------|---------|
| 单元测试覆盖率 | ≥80% | SonarQube报告 |
| API响应时间（P95） | ≤500ms | k6测试报告 |
| 页面首屏加载 | ≤2s | Lighthouse报告 |
| 系统可用性 | ≥99.9% | 试运行30天监控数据 |
| 安全漏洞 | 零高危/严重漏洞 | 渗透测试报告 |
